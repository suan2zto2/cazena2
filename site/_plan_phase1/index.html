    1단계 구현 기획 문서 계획

    작성일: 2026-05-06
    대상 구현 기간: 05/11 – 05/22 (보드 시스템)
    기준 일정: docs/schedule.md 1단계

    1. 기존 문서에서 이미 정의된 내용

    docs/systems/overview.md — 보드 시스템 섹션

    아래 항목은 overview.md에 명확히 서술되어 있어 재작성 불필요.

    항목 정의 내용
    보드 기본 크기 4×4 (16칸). 유물·패시브에 의해 확장 가능
    슬라이드 동작 지정 방향으로 모든 타일이 밀림. 인접한 동일 숫자 타일은 병합되어 두 배 값이 됨
    새 타일 생성 타이밍 슬라이드 1회 후 빈 칸 중 무작위 1칸에 생성
    새 타일 후보 값 2 또는 4
    슬라이드 횟수 제한 턴당 제한 존재 (기본값 미정)
    타일 소모 조건 카드 발동 숫자와 정확히 일치해야 소모. 상위 숫자 대체 불가
    행동 불능 조건 보드가 꽉 차서 슬라이드해도 어떤 타일도 이동하지 않는 경우
    행동 불능 효과 슬라이드와 카드 발동 모두 차단
    유효 슬라이드 기준 병합이 없더라도 타일이 물리적으로 이동했으면 유효. 카운트 소모 + 새 타일 생성 발생

    docs/index.md — 전투 시스템 보드 관련 내용

    항목 정의 내용
    보드 위기 상황 보드가 꽉 차면 슬라이드 시 새 타일이 생성되지 않음 (overview.md와 일부 상충 — 아래 미결 사항 참고)
    슬라이드의 적 카운트 영향 슬라이드는 적 행동 카운트에 영향을 주지 않음

    2. 1단계 구현 항목별 문서화 상태

    구현 항목 문서화 상태 비고
    4×4 격자 데이터 구조 및 렌더링 부분 정의 보드 크기·타일 값 목록 언급됨. 데이터 구조(배열 형태, 타일 객체 속성, 빈 칸 표현 방식)는 미정의
    상·하·좌·우 슬라이드 로직 (이동 + 병합) 규칙 수준 정의 슬라이드 방향·병합 배율 명시됨. 병합 우선순위(슬라이드 방향 기준 순서), 1회 슬라이드에서 연쇄 병합 허용 여부는 미정의
    슬라이드 후 랜덤 빈 칸에 2/4 타일 생성 부분 정의 생성 타이밍·값 후보 명시됨. 2와 4의 생성 확률 비율 미정
    턴당 슬라이드 횟수 제한 처리 규칙 언급만 제한 존재 확인됨. 기본 횟수값, 소진 시 동작(슬라이드 입력 차단 여부), 횟수 리셋 타이밍 미정
    행동 불능 판정 조건·효과 정의 조건과 효과 모두 명시됨. 판정 시점(슬라이드 시도 시점인지 슬라이드 횟수 소진 후인지), 해제 조건 미정

    3. 구현 전 반드시 확정해야 할 미결 사항

    3-A. 슬라이드 횟수 관련

    # 미결 항목 선택지 예시 영향 범위
    1 턴당 슬라이드 기본 횟수 ~~3, 4, 5 중 선택~~ → 스테이지별로 설정. 스테이지 기획 시 결정 핵심 밸런스. 게임 전체 난이도 곡선에 직결
    2 슬라이드 횟수 소진 시 동작 ~~(a) 입력 자체를 차단 / (b) 슬라이드는 가능하나 새 타일 미생성~~ → 0이 되면 슬라이드 입력 차단. 액션 버튼만 눌러야 함 구현 방식과 UX에 영향
    3 횟수 리셋 타이밍 ~~(a) 액션 버튼 누른 뒤 / (b) 새 핸드 드로우 후~~ → 새 핸드 드로우 후 리셋 해소됨
    4 패시브/유물에 의한 횟수 증감 방식 절대값(+2) vs. 배율(×1.5) 성장 시스템과의 인터페이스 설계에 영향

    3-B. 타일 생성 관련

    # 미결 항목 선택지 예시 영향 범위
    5 2 타일과 4 타일의 생성 확률 비율 ~~90%:10%, 80%:20%, 75%:25%~~ → 기본은 2 타일만 생성. 스테이지/캐릭터 특성으로 4 타일 생성 추가 가능 (초기에는 기본만) 밸런스. 스테이지·캐릭터 설계 시 확장
    6 전투 시작 시 보드 초기 상태 ~~(a) 완전 빈 보드 / (b) 2 타일 2~4개 배치 후 시작~~ → 완전 빈 보드 해소됨
    7 보드가 꽉 찬 상태에서의 슬라이드 후 타일 생성 ~~(a) 병합으로 빈 칸이 생기면 생성 / (b) 원래 빈 칸이 없으면 미생성~~ → (a) 채택. 슬라이드 후 빈 칸이 있으면 생성, 없으면 미생성. index.md 수정 완료 해소됨

    3-C. 병합 로직 관련

    # 미결 항목 선택지 예시 영향 범위
    8 1회 슬라이드에서 연쇄 병합 허용 여부 ~~(a) 표준 2048 규칙: 이미 병합된 타일은 같은 슬라이드 내에서 재병합 불가 / (b) 연쇄 병합 허용~~ → 표준 2048 규칙 채택. 연쇄 병합 불허 해소됨
    9 슬라이드 방향별 병합 처리 순서 ~~(a) 슬라이드 방향 기준 앞에서 뒤 순서로 처리 (표준 2048) / (b) 별도 규칙~~ → 표준 2048 규칙 채택 해소됨

    3-D. 행동 불능 관련

    # 미결 항목 선택지 예시 영향 범위
    10 슬라이드 불능 판정 시점 ~~(a) 슬라이드 시도 직후 / (b) 횟수 소진 시 / (c) 모든 방향 불가 시점~~ → 슬라이드 시도 직후 이동 없음 감지 시 슬라이드 입력만 차단 해소됨
    11 슬라이드 불능 해제 조건 ~~(a) 즉시 패배 / (b) 유물·패시브로 강제 해제~~ → 카드 발동으로 타일이 소모된 뒤 슬라이드 가능 상태가 되면 자동 해제. 액션 버튼으로 턴 진행 가능 해소됨
    12 슬라이드 불능 시 적 행동 처리 ~~(a) 카운트 감소 없음 / (b) 즉시 패배~~ → 액션 버튼으로 턴을 정상 진행하므로 적 카운트 정상 감소 해소됨

    3-E. 문서 간 기술 불일치 — 확인 필요

    index.md와 overview.md 사이에 아래 표현이 상충한다. 구현 전 하나로 통일해야 한다.

    항목 index.md 기술 overview.md 기술
    보드가 꽉 찼을 때 슬라이드 결과 "새 타일이 생성되지 않음" (슬라이드 자체는 가능) "슬라이드해도 어떤 타일도 이동하지 않으면 행동 불능" (조건이 다름)

    → 실제로 두 상황은 별개다: (상황 A) 보드가 꽉 찼지만 병합이 일어나는 경우 / (상황 B) 어느 방향으로도 타일 이동이 없는 경우. 두 상황을 구분하여 각각의 동작을 명시해야 한다.

    4. 작성이 필요한 문서와 섹션 계획

    우선순위 높음 — 1단계 구현에 직접 필요

    4-1. docs/systems/board.md 신규 작성 (최우선)

    기존 overview.md 보드 섹션의 규칙을 포함하되, 구현자가 코드 작성에 바로 참조할 수 있는 수준으로 상세화한다.

    작성할 섹션:

    1. 보드 데이터 구조
   - 4×4 2차원 배열 표현 방식
   - 타일 단위 데이터 (값, 위치, 이번 슬라이드에서 병합 여부 플래그)
   - 빈 칸 표현 방식 (null / 0 / 별도 타입)

2. 슬라이드 알고리즘 명세
   - 4방향별 처리 순서 정의 (예: 왼쪽 슬라이드 시 각 행을 왼쪽부터 순회)
   - 타일 이동 로직 (빈 칸 채우기 순서)
   - 병합 로직 (같은 값 감지 → 병합 → 병합 완료 플래그 설정)
   - 연쇄 병합 허용 여부 명시 [미결 #8 확정 후 작성]
   - 유효 슬라이드 판정 로직 (이동 전후 보드 상태 비교)

3. 새 타일 생성 규칙
   - 생성 트리거: 유효 슬라이드 완료 직후
   - 생성 위치: 빈 칸 목록에서 무작위 1칸
   - 생성 값: 2 또는 4 (확률 비율 [미결 #5 확정 후 작성])
   - 보드 만원 상태에서의 처리 [미결 #7 확정 후 작성]

4. 슬라이드 횟수 제한
   - 기본 횟수 값 [미결 #1 확정 후 작성]
   - 횟수 카운터 감소 타이밍 (유효 슬라이드 시에만 감소 vs. 시도마다 감소)
   - 횟수 소진 시 입력 처리 [미결 #2 확정 후 작성]
   - 횟수 리셋 타이밍 [미결 #3 확정 후 작성]

5. 행동 불능 판정
   - 판정 조건: 4방향 모두에서 유효 슬라이드가 불가능한 상태
   - 판정 시점 [미결 #10 확정 후 작성]
   - 행동 불능 상태의 효과 (슬라이드 입력 차단, 카드 발동 차단)
   - 해제 조건 [미결 #11 확정 후 작성]
   - 행동 불능 시 적 행동 처리 [미결 #12 확정 후 작성]

6. 전투 시작 시 보드 초기화
   - 초기 보드 상태 [미결 #6 확정 후 작성]
   - 초기화 순서 (빈 보드 → 초기 타일 배치 → 핸드 드로우와의 순서 관계)

7. 예외 케이스 목록
   - index.md / overview.md 불일치 항목 통합 정리 [미결 3-E 확정 후 작성]
   - 보드 꽉 참 + 슬라이드 시 병합 발생 케이스
   - 보드 꽉 참 + 슬라이드 시 병합도 없는 케이스 (행동 불능)
