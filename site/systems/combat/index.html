    전투 시스템 상세 명세

    기준 문서: docs/systems/overview.md §전투 시스템, docs/index.md §코어 루프
    관련 구현 일정: docs/schedule.md 2단계 (05/25–05/29), 3단계 (06/01–06/05)
    이 문서의 액션 버튼 처리 흐름(§4) 단계 번호가 카드·적 AI 구현의 실행 순서 기준이다.

    1. 개요 — 코어 루프 흐름도

    전투는 "슬라이드 페이즈 → 액션 버튼 → 카드 실행 → 핸드 교체 → 적 행동"을 적 전멸까지 반복하는 구조다. 슬라이드는 적 카운트에 영향을 주지 않으며, 액션 버튼을 누르는 시점에만 적 카운트가 감소한다.

    [전투 시작]
    ↓
전투 시작 초기화 (§2)
  보드 초기화 → 선제 타일 패시브 적용 → 덱 셔플 → 5장 드로우 → slideCount 초기화
    ↓
┌─────────────────────────────────────────┐
│  [슬라이드 페이즈] (§3)                  │
│  slideCount > 0이고 isSlideBlocked = false│
│  이면 슬라이드 입력 수신                  │
│    ↓ (슬라이드 or 슬라이드 횟수 소진)    │
│  [액션 버튼 입력]                        │
│    ↓                                    │
│  액션 버튼 처리 흐름 (§4) 단계 1~10 순행  │
│    ↓                                    │
│  전투 종료 판정 (§6)                     │
│    → 승리 조건 충족: 전투 종료 (§6-1)    │
│    → 패배 조건 충족: 전투 종료 (§6-2)    │
│    → 미충족: 슬라이드 페이즈로 복귀       │
└─────────────────────────────────────────┘

    상태 전이 요약:

    현재 상태 가능한 액션 다음 상태
    슬라이드 페이즈 슬라이드 (slideCount > 0, 불능 아님) 슬라이드 페이즈 (slideCount 감소)
    슬라이드 페이즈 액션 버튼 액션 버튼 처리 중
    액션 버튼 처리 중 (자동 진행) 슬라이드 페이즈 or 전투 종료
    전투 종료 (승리) (자동) 카드 보상 선택 화면
    전투 종료 (패배) (자동) 챕터 재시작 or 세이브 포인트 복귀

    2. 전투 시작 초기화 순서

    전투 진입 시 아래 순서를 반드시 지킨다. 각 단계가 완료된 후 다음 단계로 진행한다.

    [전투 시작 초기화]

단계 1. 출전 캐릭터 확정
        - 전투 진입 화면에서 플레이어가 최대 3인을 선택
        - 미출전 캐릭터의 카드는 이번 전투 덱에 포함하지 않는다
        - 확정된 출전 캐릭터 목록을 CombatState에 기록

단계 2. 보드 초기화
        - board[0..3][0..3] 전체를 0으로 초기화
        - isSlideBlocked = false

단계 3. 패시브 "선제 타일" 적용 (보유 시)
        - 빈 칸(값 0인 칸) 목록에서 무작위 2칸을 선택
        - 선택된 2칸에 값 4인 타일을 배치
        - 보드 초기화(단계 2) 완료 후에만 실행. 드로우 전에 완료해야 한다

단계 4. 패시브 "카운트 지연" 적용 (보유 시)
        - 이번 전투에 등장하는 모든 적의 초기 카운트 값에 +1 적용
        - 적 데이터를 메모리에 로드한 직후 카운트 초기값 조정

단계 5. 덱 구성 및 셔플
        - 출전 캐릭터(단계 1)의 카드를 합산하여 공용 카드 풀 생성
        - 풀 전체를 셔플하여 대기덱으로 설정
        - 핸드덱 = 빈 배열, 소모덱 = 빈 배열

단계 6. 핸드 드로우
        - 대기덱에서 5장을 드로우하여 핸드덱 구성
        - 대기덱이 5장 미만인 경우 처리는 card.md §드로우 규칙을 따른다

단계 7. slideCount 초기화
        - slideCount = maxSlides (스테이지 데이터에서 주입된 값)
        - board.md §5-4 참조: 새 핸드 드로우 완료 후 리셋. 전투 시작 시 핸드 드로우(단계 6) 완료가 이 조건을 충족한다

단계 8. 슬라이드 페이즈 진입
        - 초기화 완료. 플레이어 입력 수신 시작

    초기화 완료 후 보드 상태 예시 (선제 타일 패시브 적용 시):

    [ 초기 보드 — 선제 타일 2개 무작위 배치 ]

     col 0  col 1  col 2  col 3
row 0 [  0] [  0] [  4] [  0]
row 1 [  0] [  0] [  0] [  0]
row 2 [  0] [  4] [  0] [  0]
row 3 [  0] [  0] [  0] [  0]

slideCount = maxSlides (예: 5)
핸드: [카드A(16), 카드B(8), 카드C(32), 카드D(4), 카드E(16)]

    3. 슬라이드 페이즈

    슬라이드 페이즈는 플레이어가 액션 버튼을 누르기 전까지 반복 가능한 구간이다.

    3-1. 슬라이드 입력 수신 조건

    슬라이드 입력을 받아들이려면 아래 두 조건을 모두 충족해야 한다.

    조건 충족 기준
    slideCount > 0 남은 슬라이드 횟수가 1 이상
    isSlideBlocked = false 4방향 중 유효한 슬라이드 방향이 1개 이상 존재

    두 조건 중 하나라도 미충족이면 슬라이드 UI를 비활성화하고 입력을 차단한다. 차단된 상태에서도 액션 버튼은 항상 조작 가능하다.

    [미결 B-4] 액션 버튼 비활성화 조건이 미확정이다.
    선택지:
    (a) 항상 활성 — 핸드가 0장이거나 매칭 카드가 없어도 누를 수 있다. 구현이 단순하나 플레이어가 실수로 턴을 낭비할 수 있다.
    (b) 핸드 0장일 때 비활성 — 드로우가 불가한 극단 상황에서만 차단한다.
    (c) 매칭 카드 0장일 때 비활성 — 실행될 카드가 없으면 누를 수 없다. 전략적 실수를 방지하지만 의도적 "빈 턴 소비"가 불가능해진다.

    미결 확정 전 구현은 (a) 항상 활성으로 임시 처리한다.

    3-2. 슬라이드 처리 순서

    슬라이드 입력이 수신되면 아래 순서를 실행한다. 상세 알고리즘은 board.md §3을 따른다.

    슬라이드 방향 입력 수신 (상·하·좌·우)
  ↓
슬라이드 연산 수행 (board.md §3-2)
  ↓
유효 슬라이드 판정 (board.md §3-4)
  → 유효: slideCount -= 1
           새 타일 생성 (board.md §4)
           isSlideBlocked 재검사 (board.md §6-2)
  → 무효: slideCount 변경 없음
           슬라이드 불능 감지 루틴 실행 (board.md §6)
  ↓
슬라이드 페이즈 유지 (액션 버튼 대기)

    3-3. 슬라이드 페이즈 종료 조건

    다음 중 하나가 발생하면 슬라이드 페이즈를 종료하고 액션 버튼 처리로 진입한다.

    • 플레이어가 액션 버튼을 누른다
    • slideCount가 0이 되어 슬라이드 입력이 차단되고 플레이어가 액션 버튼을 누른다
    • isSlideBlocked = true가 되어 슬라이드 입력이 차단되고 플레이어가 액션 버튼을 누른다

    슬라이드 페이즈는 액션 버튼 입력 없이 자동으로 종료되지 않는다. 플레이어가 능동적으로 액션 버튼을 눌러야 한다.

    4. 액션 버튼 처리 흐름

    액션 버튼이 눌리면 아래 단계를 순서대로 실행한다. 각 단계 번호가 카드 시스템·적 AI 구현의 처리 순서 기준이다.

    액션 버튼 눌림
  ↓
단계 1. [중복 숫자 카드 선택 UI]
단계 2. [핸드·보드 매칭 검사]
단계 3. [매칭 카드 순차 실행 + 파티 연계 판정]
단계 4. [전투 종료 조건 중간 확인]
단계 5. [핸드 전체 소모덱 이동]
단계 6. [재드로우]
단계 7. [slideCount 리셋]
단계 8. [모든 적 카운트 -1]
단계 9. [카운트 0인 적 행동 처리]
단계 10.[전투 종료 조건 최종 확인 → 슬라이드 페이즈 복귀]

    단계 1 — 중복 숫자 카드 선택 UI

    핸드 카드 중 발동 숫자가 동일한 카드가 2장 이상 존재하고, 보드에 해당 숫자 타일이 존재하는 경우, 플레이어가 어느 카드를 발동할지 선택하는 단계다.

    • 중복이 없으면 이 단계는 건너뛴다.
    • 발동 숫자별로 독립적으로 처리한다. 예: 핸드에 발동 숫자 16짜리 카드가 2장이고 보드에 16 타일이 1개인 경우, 두 카드 중 하나를 선택한다.
    • 보드에 해당 숫자 타일이 없는 경우 선택 UI 없이 둘 다 발동 불가 처리된다.

    [미결 B-3] 중복 숫자 카드 선택 UI 발동 시점이 미확정이다.
    (a) 액션 버튼을 누르기 전에 핸드에서 항상 선택 가능 — 사전 계획 강조. 액션 버튼 시점에는 이미 선택이 완료된 상태.
    (b) 액션 버튼 누른 뒤 매칭 발견 시 팝업 — 확인 절차 강조. 단계 1에서 인터럽트 방식으로 처리.

    미결 확정 전까지는 (b) 액션 버튼 후 팝업 방식으로 구현한다. 팝업 중 다른 입력은 차단한다.

    단계 2 — 핸드·보드 매칭 검사

    핸드의 각 카드에 대해 보드에 발동 숫자와 동일한 타일이 존재하는지 확인한다.

    매칭 검사 규칙:

    • 카드의 발동 숫자와 보드 타일의 값이 정확히 일치해야 한다. 상위 숫자 타일로의 대체 발동은 허용하지 않는다. (예: 발동 숫자 16인 카드는 보드에 16 타일이 있을 때만 매칭. 32 타일이 있어도 매칭되지 않는다.)
    • 단계 1에서 중복 선택이 이루어진 경우, 선택되지 않은 카드는 해당 숫자에 대한 매칭에서 제외된다.
    • 매칭 결과를 "발동 대상 카드 목록"과 "미발동 카드 목록"으로 분리한다.

    동일 숫자 타일 다수 존재 시 소모 위치:

    보드에 동일 숫자 타일이 여러 개 존재하는 경우, 소모할 타일의 위치를 결정해야 한다.

    [미결 B-2] 동일 숫자 타일 다수 존재 시 소모 위치 선택 방식이 미확정이다.
    (a) 자동 선택 — 행 인덱스 오름차순, 같은 행에서는 열 인덱스 오름차순으로 첫 번째 발견된 타일을 소모한다. 구현 단순.
    (b) 플레이어가 소모할 타일 위치를 직접 선택 — 전략성은 높지만 추가 선택 UI 필요.

    미결 확정 전까지는 (a) 자동 선택 (행·열 오름차순 첫 번째) 으로 구현한다.

    단계 3 — 매칭 카드 순차 실행 + 파티 연계 판정

    발동 대상 카드 목록을 정해진 순서에 따라 하나씩 실행한다. 각 카드를 실행할 때마다 파티 연계 판정을 수행한다.

    카드 실행 순서:

    [미결 B-1] 카드 실행 순서가 미확정이다.
    (a) 핸드 왼쪽(낮은 인덱스) 순 — 핸드 배열에서 인덱스 0부터 순서대로 실행.
    (b) 발동 숫자 오름차순 — 낮은 숫자 카드부터 실행. 약한 카드가 먼저 효과를 적용.
    (c) 발동 숫자 내림차순 — 높은 숫자 카드부터 실행. 강한 카드가 먼저 효과를 적용.

    B-6(연계 판정 시점)과 연동된다. 미결 확정 전까지는 (a) 핸드 왼쪽(낮은 인덱스) 순으로 구현한다.

    카드 1장 실행 시 처리 순서:

    현재 카드 선택 (결정된 실행 순서에 따라)
  ↓
파티 연계 판정 (§5 참조)
  → 연계 성립: 연계 보너스 효과 적용 (§5-3)
  → 연계 불성립: 넘어감
  ↓
보드에서 해당 발동 숫자 타일 소모 (단계 2에서 결정된 위치)
  ↓
카드 효과 적용 (공격·회복·버프·디버프 등 card.md §효과 유형 참조)
  ↓
전투 종료 조건 즉시 확인
  → 모든 적 처치: 단계 4 (중간 확인)로 이동
  → 파티 전원 사망: 단계 4 (중간 확인)로 이동
  → 미충족: 다음 카드로 진행
  ↓
발동 대상 카드 목록의 다음 카드로 반복

    "이미 선택한 카드의 효과 완전 적용" 원칙:

    카드 효과 적용 도중 적이 사망하거나 파티원이 사망하더라도, 현재 실행 중인 카드의 효과는 끝까지 적용한다. 효과 완전 적용 후 다음 카드로 진행하기 전에 종료 조건을 확인한다. (overview.md §전투 시스템 §예외 처리 기준)

    단계 4 — 전투 종료 조건 중간 확인

    단계 3의 모든 카드 실행이 끝난 직후 전투 종료 여부를 확인한다.

    조건 처리
    모든 적이 처치된 상태 전투 종료 — 승리 처리 (§6-1)로 이동. 단계 5 이후를 실행하지 않는다
    파티 전원이 사망한 상태 전투 종료 — 패배 처리 (§6-2)로 이동. 단계 5 이후를 실행하지 않는다
    미충족 단계 5로 진행

    카드 실행 도중 적 전멸과 파티 전원 사망이 동시에 충족되는 상황(예: 파티원이 적의 반격 효과로 동시 사망)은 승리 우선으로 처리한다. 이 규칙이 적용될 수 있는 카드 효과 설계 시 유의한다.

    단계 5 — 핸드 전체 소모덱 이동

    발동된 카드와 발동되지 않은 카드 모두를 소모덱으로 이동한다.

    • 핸드덱 배열 전체를 소모덱 배열에 추가한다.
    • 핸드덱 배열을 빈 배열로 초기화한다.
    • 사망한 캐릭터의 카드도 동일하게 소모덱으로 이동한다. (발동 불가 상태는 유지)

    단계 6 — 재드로우

    대기덱에서 새 핸드를 드로우한다. 상세 규칙은 card.md §드로우 규칙을 따른다.

    • 기본 드로우 장수: 5장
    • 대기덱이 부족하면 소모덱을 셔플하여 대기덱으로 전환 후 드로우를 계속한다.
    • 대기덱과 소모덱 모두 0장이면 드로우하지 않는다. 핸드덱은 빈 채로 다음 단계를 진행한다.

    단계 7 — slideCount 리셋

    재드로우(단계 6) 완료 후 slideCount를 초기화한다.

    slideCount = maxSlides
    • 리셋 타이밍은 반드시 핸드 드로우 완료 후다. (board.md §5-4 기준)
    • 패시브·유물에 의해 maxSlides가 변경된 경우 변경된 값으로 리셋한다.

    단계 8 — 모든 적 카운트 -1

    전장에 생존한 모든 적의 행동 카운트를 1 감소시킨다.

    for each 생존 적 in 적 목록:
    적.카운트 -= 1
    • 이미 사망한 적(체력 0 이하)은 카운트 감소 대상에서 제외한다.
    • 카운트가 이미 0인 적은 이 단계에서 음수가 되지 않는다. 카운트는 최솟값 0으로 클램프한다. (단계 9에서 즉시 처리됨)

    단계 9 — 카운트 0인 적 행동 처리

    단계 8 완료 후, 카운트가 0인 적을 적 목록 위→아래(인덱스 오름차순) 순서로 순차 처리한다.

    적 1개 처리 순서:

    카운트 == 0인 적 확인
  ↓
예고 행동 실행 (enemy.md §행동 유형 참조)
  ↓
행동 적용 결과로 파티원 사망 확인
  → 파티 전원 사망: 즉시 단계 10으로 이동 (남은 적 행동은 실행하지 않음)
  → 생존자 있음: 계속 진행
  ↓
해당 적의 카운트 리셋 (enemy.md §행동 카운트 처리 규칙 참조)
  ↓
다음 카운트 0인 적 처리
    • 복수의 적이 동시에 카운트 0이 되는 경우: 적 목록의 인덱스 순서(위→아래)로 순차 실행한다. 한 적의 행동이 완전히 완료된 후 다음 적의 행동을 실행한다.
    • 카운트 0인 적이 다른 적의 행동 중 카드 효과로 처치된 경우: 처치된 적의 예고 행동은 취소한다. (overview.md §적 AI 시스템 §예외 처리 기준)

    단계 10 — 전투 종료 조건 최종 확인 → 슬라이드 페이즈 복귀

    단계 9 완료 후 전투 종료 여부를 최종 확인한다.

    조건 처리
    모든 적이 처치된 상태 전투 종료 — 승리 처리 (§6-1)
    파티 전원이 사망한 상태 전투 종료 — 패배 처리 (§6-2)
    미충족 슬라이드 페이즈로 복귀

    5. 파티 연계 판정

    5-1. 연계 조건

    아래 두 조건을 모두 충족하면 연계가 성립한다.

    조건 판정 기준
    다른 캐릭터 카드 직전에 실행된 카드의 소유 캐릭터 ID와 현재 실행할 카드의 소유 캐릭터 ID가 다르다
    슬라이드 없음 직전 카드 실행 이후 슬라이드가 발생하지 않았다 (같은 액션 버튼 처리 사이클 내에서 연속 실행)
    • "직전 카드"는 단계 3 내에서 바로 이전에 실행된 카드를 의미한다. 이전 턴의 카드와는 비교하지 않는다.
    • 첫 번째로 실행되는 카드는 비교 대상이 없으므로 연계를 판정하지 않는다.

    5-2. 연계 판정 시점

    [미결 B-6] 연계 판정 시점이 미확정이다.
    (a) 카드 실행 처리 시 직전 카드와 비교 — 단계 3에서 각 카드를 실행할 때마다 직전 카드와 실시간으로 비교한다. B-1(실행 순서)과 연동되며, 실행 순서에 따라 연계 결과가 달라진다.
    (b) 플레이어가 카드 순서를 지정한 시점에 미리 판정 — 액션 버튼 전 또는 단계 1에서 순서를 지정할 때 연계 여부를 미리 표시한다. 구현 복잡도가 높으나 플레이어 가시성이 좋다.

    B-1과 연동하여 함께 결정한다. 미결 확정 전까지는 (a) 카드 실행 처리 시 직전 카드와 비교로 구현한다.

    5-3. 연계 보너스 효과

    연계가 성립하면 해당 카드 실행 직전에 연계 보너스 효과를 적용한다.

    [미결 B-5] 연계 보너스 수치가 미확정이다.
    추가 광역 피해의 기본값과 계산 기준을 결정해야 한다.
    선택지:
    (a) 고정값 — 예: 전체 적에게 피해 5. 구현 단순. 후반부 스케일 문제 가능.
    (b) 직전 카드 기본 피해의 N% — 예: 직전 카드 피해의 30%. 스케일 자동 조정. 계산 기준 정의 필요.
    (c) 두 카드 발동 숫자 합산 기준 고정값 — 예: (카드A 발동 숫자 + 카드B 발동 숫자) × 계수. 고숫자 연계에 보상 집중.

    미결 확정 전까지는 플레이스홀더로 CHAIN_BONUS_DAMAGE = TBD를 상수로 정의하고 구현한다. 패시브 "연계 전문" 보유 시 이 값에 ×1.5를 적용한다.

    연계 보너스 형태: 전체 적에게 추가 광역 피해 적용.

    5-4. 이중 연계

    3인 파티가 A → B → C 순서로 연속 발동하는 경우 이중 연계가 성립한다.

    • A→B 실행 시 연계 보너스 1회 적용
    • B→C 실행 시 연계 보너스 1회 추가 적용
    • 연계 보너스가 중첩되어 총 2회 적용된다

    이중 연계 시각 예시:

    핸드: [캐릭터A 카드(16), 캐릭터B 카드(16), 캐릭터C 카드(32)]
보드: 16 타일 2개, 32 타일 1개 존재

실행 순서 (핸드 인덱스 순 기준):
  1. 캐릭터A 카드(16) 실행 — 연계 없음 (첫 카드)
  2. 캐릭터B 카드(16) 실행 — 연계 성립 (A≠B) → 연계 보너스 1회
  3. 캐릭터C 카드(32) 실행 — 연계 성립 (B≠C) → 연계 보너스 1회
총 연계 보너스: 2회 적용

    5-5. 연계 불성립 케이스

    상황 판정
    같은 캐릭터 카드를 연속 실행 연계 불성립. 보너스 없이 일반 발동
    사망한 캐릭터 카드 — 직전에 실행된 후 다른 캐릭터 카드 실행 사망한 캐릭터의 카드는 실행되지 않으므로 연계 판정에서 "직전 카드"로 간주하지 않는다
    핸드에 단 1장만 매칭 직전 카드가 없으므로 연계 불성립

    6. 전투 종료 판정

    6-1. 승리 조건 및 처리 흐름

    조건: 전장의 모든 적의 체력이 0 이하가 된다.

    확인 시점: 단계 3(카드 실행 중 매 카드 실행 후), 단계 4(카드 실행 전체 완료 후), 단계 10(적 행동 처리 완료 후).

    승리 처리 순서:

    전투 승리 확정
  ↓
1. 현재 진행 중인 카드 효과가 있다면 완전히 적용 완료
2. 전투 상태를 "종료" 로 설정 → 슬라이드·액션 버튼 입력 차단
3. 승리 연출 재생 (연출 규칙은 ux.md 참조)
4. 카드 보상 선택 화면 진입
   - 덱 전체에서 무작위 3장을 후보로 제시 (후보 선정 규칙은 card.md 참조)
   - 플레이어가 1장을 선택하여 덱에 추가
   - 선택 취소 또는 건너뛰기 가능 여부는 미정
5. 거점 귀환
   - 보드 상태·전투 상태 초기화
   - 챕터 내 성장(패시브·유물·카드) 유지

    6-2. 패배 조건 및 처리 흐름

    조건: 출전 파티 전원(최대 3인)의 체력이 0 이하가 된다.

    확인 시점: 단계 3(카드 실행 중 매 카드 실행 후), 단계 4(카드 실행 전체 완료 후), 단계 9(적 행동 처리 중 매 행동 후), 단계 10(적 행동 처리 완료 후).

    패배 처리 순서:

    전투 패배 확정
  ↓
1. 현재 진행 중인 카드 효과가 있다면 완전히 적용 완료
   (단계 9 중 패배 확정 시에는 남은 적 행동을 실행하지 않는다 — §4-9 참조)
2. 전투 상태를 "종료"로 설정 → 모든 입력 차단
3. 패배 연출 재생 (연출 규칙은 ux.md 참조)
4. 선택지 제시:
   (a) 챕터 재시작 — 챕터 시작 상태로 돌아간다. 챕터 내 획득 카드·패시브·유물 모두 초기화
   (b) 세이브 포인트 복귀 — 가장 최근 세이브 포인트 상태로 복귀
   (세이브 포인트 구조는 progression.md 참조)

    6-3. 카드 실행 중 종료 조건 충족

    카드 실행(단계 3) 도중 종료 조건이 충족되는 세 가지 케이스와 처리 방식:

    케이스 상황 처리
    적 전멸 카드 효과로 마지막 적이 처치됨 현재 카드의 효과 완전 적용 완료 후 → 단계 4로 이동. 남은 발동 대상 카드는 실행하지 않는다
    파티 전멸 카드의 자기 피해·반격 효과 등으로 파티 전원 사망 현재 카드의 효과 완전 적용 완료 후 → 단계 4로 이동
    동시 충족 같은 카드 효과로 적 전멸과 파티 전멸이 동시에 발생 승리 우선 처리. 단계 4에서 승리로 판정

    7. 예외 케이스

    케이스 1 — 핸드 0장 상태에서 액션 버튼

    대기덱과 소모덱이 모두 소진되어 핸드에 카드가 없는 상태에서 액션 버튼이 눌린 경우.

    처리: - 단계 1~3을 건너뛴다 (매칭 카드 없음, 실행 카드 없음). - 단계 4: 종료 조건 미충족으로 판단 (승리·패배 조건은 체력 기준이므로 카드 미실행과 무관). - 단계 5: 핸드덱이 이미 비어 있으므로 소모덱 이동 대상 없음. - 단계 6: 대기덱·소모덱 모두 0장이면 드로우 없이 진행. - 단계 7~10: 정상 실행. 적 카운트는 정상적으로 감소한다.

    이 케이스는 플레이어가 아무런 행동 없이 턴을 소비하는 것과 동일하다. 전략적으로 불리하지만 규칙상 허용된다. 미결 B-4 확정 시 이 케이스에서 액션 버튼을 비활성화할지 결정한다.

    케이스 2 — 카드 발동 중 파티원 사망 처리

    단계 3 실행 중 카드 효과로 파티원 1인이 사망(체력 0 이하)하는 경우.

    처리: - 현재 카드의 효과 완전 적용 후 계속 진행한다. - 사망한 캐릭터의 이후 카드 실행은 건너뛴다. 실행 순서에서 사망한 캐릭터 소유 카드가 남아 있더라도 발동하지 않는다. (사망 캐릭터 카드는 발동 불가 — overview.md §카드 시스템 기준) - 사망한 캐릭터가 직전 카드 소유자인 경우, 다음 카드의 연계 판정에서 "직전 카드" 소유자를 사망 캐릭터가 아닌 그 이전의 마지막 실행된 생존 캐릭터 카드 기준으로 판정한다.

    케이스 3 — 복수 적 동시 카운트 0 상세 처리

    단계 8에서 복수의 적이 동시에 카운트 0이 된 경우.

    처리: 적 목록의 인덱스 오름차순(화면 위→아래 순서)으로 순차 실행.

    예시:
적 목록 [인덱스 0: 고블린, 인덱스 1: 오크, 인덱스 2: 드래곤]

단계 8 후 고블린(0)·드래곤(2) 카운트 동시 0 도달 시:
  1. 고블린 행동 실행 → 카운트 리셋
  2. 파티 전멸 확인 → 미충족 시 계속
  3. 드래곤 행동 실행 → 카운트 리셋
  4. 파티 전멸 확인
    • 한 적의 행동 실행 도중 파티 전원이 사망하면 남은 적의 행동은 실행하지 않고 단계 10으로 이동한다.
    • 한 적의 행동 실행 도중 해당 적이 카드 반격 효과 등으로 처치되면, 이미 시작된 해당 적의 행동은 완전 적용 후 종료한다.

    케이스 4 — slideCount 0 상태에서 isSlideBlocked 해제

    카드 발동으로 타일이 소모되어 슬라이드 불능(isSlideBlocked = true)이 해제되더라도, slideCount가 이미 0인 경우 슬라이드 입력은 여전히 차단된다. (board.md §케이스 D 기준)

    이 상태는 슬라이드 불능이 아니라 횟수 소진으로 인한 차단이다. 슬라이드 UI에서 두 상태를 구분하여 표시해야 한다.

    상태 isSlideBlocked slideCount 슬라이드 입력
    정상 슬라이드 가능 false > 0 가능
    횟수 소진 차단 false 0 차단
    불능 차단 true > 0 차단
    횟수 소진 + 불능 true 0 차단

    케이스 5 — 전투 시작 시 덱이 5장 미만인 경우

    전투 시작 초기화(§2 단계 6)에서 대기덱 카드 수가 5장 미만인 경우의 처리는 card.md §드로우 규칙을 따른다. 전투 초기화 단계에서는 소모덱이 빈 상태이므로, 있는 장수만 드로우하고 시작한다.

    케이스 6 — 보스 페이즈 전환이 적 행동 실행 중 발생

    단계 9에서 카운트 0인 보스의 행동을 실행하는 도중, 해당 행동 내에서 보스 체력 구간이 변경되어 페이즈 전환 조건이 충족되는 경우.

    처리: 현재 행동을 완전히 실행 완료한 후 페이즈 전환을 처리한다. 페이즈 전환은 현재 행동 실행 도중 인터럽트하지 않는다. 페이즈 전환 상세는 enemy.md §보스 페이즈 시스템을 따른다.
