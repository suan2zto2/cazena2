    카드 시스템

    📄 docs/systems/card.md

    참고 문서: 게임 세부 시스템 개요 §카드 시스템, 게임 개요 §스킬 카드와 발동 조건
    관련 문서: 보드 시스템, 전투 시스템 (작성 예정)

    1. 개요

    카드 시스템은 출전 캐릭터의 스킬 카드를 대기덱·핸드덱·소모덱 세 영역으로 통합 관리하는 구조다. 각 카드에는 보드에서 소모해야 하는 발동 숫자(4·8·16·32·64·128)가 지정되어 있으며, 숫자가 높을수록 강력한 효과를 제공한다.

    카드 시스템의 흐름 요약

    전투 시작
    └─ 선택 캐릭터의 카드를 셔플 → 대기덱 구성
           └─ 5장 드로우 → 핸드덱 구성
                  ↓
           [슬라이드 페이즈] 보드에 타일 준비
                  ↓
           [카드 선택] 같은 숫자 카드 여러 장이면 원하는 카드 선택
                  ↓
           [액션 버튼]
             ├─ 매칭된 카드: 타일 소모 → 효과 발동
             └─ 미매칭 카드: 효과 없이 대기
                  ↓
           핸드 전체 소모덱으로 이동 → 대기덱에서 재드로우
                  ↓
           [반복] 전투 종료 → 보상 3장 중 1장 선택

    2. 카드 데이터 구조

    2-1. Card 구조체

    필드명 타입 설명
    id string 카드 고유 식별자 (예: "char_a_attack_01")
    name string 카드 표시 이름 (예: "참격")
    ownerCharacterId string 이 카드를 소유하는 캐릭터 ID. 소유 캐릭터 식별에 사용
    triggerNumber int 발동에 필요한 보드 타일 숫자. 허용값: 4·8·16·32·64·128
    grade CardGrade 발동 숫자로 결정되는 등급 열거형 (§2-3 참고)
    effectType CardEffectType 카드 효과 분류 열거형 (§2-2 참고)
    effectParams EffectParams 효과 실행에 필요한 파라미터 묶음 (§2-2 참고)

    제약 조건

    • triggerNumber는 반드시 {4, 8, 16, 32, 64, 128} 집합에 속해야 한다.
    • upgradedTriggerNumber가 존재하는 경우, 반드시 triggerNumber보다 작아야 한다.
    • upgradeLevel이 0이면 upgradedTriggerNumber는 null이다.
    • ownerCharacterId는 현재 전투에 출전한 캐릭터의 ID와 일치해야 한다. 미출전 캐릭터의 카드는 덱에 포함되지 않는다.

    런타임에서의 발동 숫자 결정

    effectiveTriggerNumber = (upgradeLevel >= 1 && upgradedTriggerNumber != null)
    ? upgradedTriggerNumber
    : triggerNumber

    2-2. 효과 타입 열거형 (CardEffectType)

    [미결 A-1] 카드 효과 타입의 범위가 확정되지 않았습니다.
    - (a) 공격·회복·버프·디버프 4종으로 고정
    - (b) 실드(Shield)·드로우(Draw)·상태이상(StatusEffect) 등 추가 타입 포함한 확장 목록

    아래 표는 (a) 기준 4종을 기본으로 작성하되, 확장 타입 후보를 함께 나열합니다. 확정 시 EffectParams 구조체도 함께 갱신해야 합니다.

    기본 4종 (확정)

    CardEffectType 값 설명 주요 EffectParams 필드
    ATTACK 적 단일 또는 전체에 피해를 입힌다 damage: int, targetType: TargetType
    HEAL 아군 단일 또는 전체의 HP를 회복한다 healAmount: int, targetType: TargetType
    BUFF 아군에게 유리한 상태 효과를 부여한다 buffId: string, duration: int, targetType: TargetType
    DEBUFF 적에게 불리한 상태 효과를 부여한다 debuffId: string, duration: int, targetType: TargetType

    확장 타입 후보 (A-1 미결 — 채택 여부 확정 필요)

    CardEffectType 후보값 설명 비고
    SHIELD 아군에게 피해를 일정량 흡수하는 방어막을 부여 방어막 수치 소진 시 자동 제거
    DRAW 대기덱에서 지정 장수를 즉시 핸드에 추가로 드로우 핸드 초과 장수 처리 규칙 별도 결정 필요
    STATUS_EFFECT 독·마비 등 지속 피해 또는 행동 제한 상태이상 부여 DEBUFF와 별도 분리 여부 결정 필요

    EffectParams 공통 구조

    EffectParams {
    targetType      : TargetType    // SINGLE_ENEMY | ALL_ENEMIES | SINGLE_ALLY | ALL_ALLIES
    damage          : int?          // ATTACK 타입 사용
    healAmount      : int?          // HEAL 타입 사용
    buffId          : string?       // BUFF 타입 사용
    debuffId        : string?       // DEBUFF 타입 사용
    duration        : int?          // BUFF·DEBUFF 지속 턴 수 (액션 버튼 횟수 기준)
    shieldAmount    : int?          // SHIELD 타입 사용 (A-1 확장 시)
    drawCount       : int?          // DRAW 타입 사용 (A-1 확장 시)
}

    duration 필드의 단위(액션 버튼 횟수 vs. 적 공격 횟수)는 미결 C-5와 연동되며, docs/systems/enemy.md 작성 시 함께 확정한다.

    2-3. 카드 등급 정의 (CardGrade)

    카드 등급은 triggerNumber로 결정되며, 강화 후 upgradedTriggerNumber가 기존 등급 경계를 넘어서도 기준 등급은 변경되지 않는다. 등급은 카드 식별 및 UI 표시 목적으로만 사용한다.

    CardGrade 값 triggerNumber 등급명 효과 규모 예시
    BASIC 4 기본 단일 약공격, 소량 방어막
    NORMAL 8 일반 단일 공격, 디버프 1개
    ENHANCED 16 강화 중간 공격, 광역 디버프, 아군 회복
    POWERFUL 32 강력 강한 광역 공격, 파티 전체 방어막
    LETHAL 64 필살 캐릭터 전용 필살기
    TRANSCENDENT 128 초월 전투 판도를 뒤집는 초강력 효과

    3. 덱 운영 구조

    3-1. DeckState 구조체

    전투 중 카드 전체의 상태를 표현하는 구조체다. 전투 시작 시 초기화되고, 전투 종료 시 소멸한다.

    DeckState {
    waitDeck    : Card[]    // 대기덱 — 드로우되지 않은 카드들. 셔플된 순서로 관리
    hand        : Card[]    // 핸드덱 — 현재 사용 가능한 카드들. 최대 5장
    discardDeck : Card[]    // 소모덱 — 발동·미발동 모두 포함해 버려진 카드들. 순서 유지
}

    [미결 A-5] 덱 전체 카드 수의 상한이 결정되지 않았습니다.
    - (a) 상한 없음 — 전투 보상으로 카드를 계속 추가할 수 있음
    - (b) 최대 N장 제한 — 상한 초과 시 기존 카드를 제거하거나 추가를 거부하는 규칙 필요

    소모덱 순환 주기와 전략적 덱 설계에 직접 영향을 줍니다. (a) 채택 시 DeckState에 상한 필드 불필요. (b) 채택 시 maxDeckSize: int 필드 추가 필요.

    3-2. 세 영역의 역할

    영역 구조체 필드 역할 드로우 가능 여부
    대기덱 (waitDeck) Card[] 아직 드로우되지 않은 카드 풀. 셔플된 상태로 유지되며 맨 앞 카드부터 드로우됨 O
    핸드덱 (hand) Card[] 현재 플레이어가 운용 중인 카드. 액션 버튼 발동의 대상이 되는 영역 X
    소모덱 (discardDeck) Card[] 사용 후 버려진 카드. 대기덱이 소진되면 전체 셔플 후 대기덱으로 전환됨 X (직접 접근 불가)

    3-3. 영역 간 이동 규칙

    아래 표는 카드가 한 영역에서 다른 영역으로 이동하는 모든 시점과 조건을 정의한다.

    이동 방향 트리거 조건 비고
    대기덱 → 핸드덱 드로우 대기덱에 1장 이상 존재 맨 앞 카드를 꺼내 핸드덱 끝에 추가
    소모덱 → 대기덱 셔플 대기덱이 0장이고 드로우가 필요한 시점 소모덱 전체를 무작위 순서로 재배열 후 대기덱으로 전환
    핸드덱 → 소모덱 액션 버튼 이후 핸드에 카드가 존재 발동 여부와 무관하게 핸드 전체 이동. 이동 순서는 핸드 내 인덱스 순서

    3-4. 전투 시작 초기화 순서

    1. 출전 캐릭터(최대 3인)의 카드를 전부 수집
2. 수집된 카드 전체를 waitDeck에 배치
3. waitDeck을 무작위 순서로 셔플
4. waitDeck에서 최대 5장 드로우 → hand에 배치
   └─ 드로우 상세 규칙은 §4 드로우 규칙 참고
5. discardDeck은 빈 배열로 초기화

    4. 드로우 규칙

    4-1. 드로우 트리거 2종

    드로우는 다음 두 시점에만 발생한다.

    트리거 시점 목표 드로우 수
    전투 시작 드로우 전투 초기화 완료 직후 (§3-4 4번 단계) 5장
    액션 버튼 이후 드로우 핸드 전체가 소모덱으로 이동한 직후 5장

    두 트리거 모두 동일한 드로우 함수(drawCards(count: 5))를 호출한다.

    4-2. 드로우 함수 동작 (drawCards(count))

    function drawCards(count: int):
    remaining = count

    while remaining > 0:
        if waitDeck.length == 0:
            if discardDeck.length == 0:
                break  // §4-4 예외 처리 적용
            else:
                shuffleDiscardIntoDeck()  // §4-3 셔플 처리

        card = waitDeck.shift()  // 맨 앞 카드 꺼내기
        hand.push(card)
        remaining -= 1

    4-3. 대기덱 소진 시 셔플 처리

    [미결 A-3] 셔플 타이밍이 확정되지 않았습니다.
    - (a) 대기덱이 0장이 되는 순간 즉시 셔플 — 드로우 필요 여부와 무관하게 소진과 동시에 셔플
    - (b) 드로우를 시도하는 시점에 대기덱이 0이면 그때 셔플 — 표준적인 TCG 방식

    현재 문서는 (b) 기준으로 작성되어 있습니다. (a) 채택 시 drawCards 함수 외부에 소진 감지 훅 추가 필요.

    shuffleDiscardIntoDeck() 처리 순서:

    1. discardDeck 전체를 waitDeck으로 이동
2. waitDeck을 무작위 순서로 셔플
3. discardDeck을 빈 배열로 초기화

    4-4. 전투 시작 드로우 시 대기덱 5장 미만 처리

    [미결 A-2] 전투 시작 시 대기덱이 5장 미만인 경우의 처리가 확정되지 않았습니다.
    - (a) 있는 장수를 먼저 드로우한 뒤, 소모덱을 셔플하여 대기덱을 보충하고 나머지를 드로우
    - (b) 있는 장수만 드로우하고 셔플 없이 전투 시작 (핸드가 5장 미만인 채로 시작 가능)

    전투 시작 시 소모덱은 빈 상태(§3-4 5번)이므로, 대기덱이 5장 미만인 상황은 전투 보상으로 카드 수가 적은 초반에만 발생합니다. (a)는 사실상 초반에 효과가 없으며 (b)와 동일하게 동작합니다. 이 시나리오 자체가 발생 불가능한지 최소 덱 구성 규칙으로 방지할 것인지도 함께 결정 필요.

    4-5. 예외: 대기덱·소모덱 모두 0장

    drawCards 함수 내에서 waitDeck.length == 0이고 discardDeck.length == 0이면 드로우를 중단한다.

    • 핸드덱은 현재까지 드로우된 장수만큼만 유지된다 (0장일 수 있음).
    • 0장 핸드 상태에서도 액션 버튼은 눌를 수 있다. 발동되는 카드가 없으므로 효과 없이 핸드 교체(이미 비어있음) → 적 카운트 -1만 처리된다.
    • 이 상태는 덱이 매우 작고 순환이 완료된 경우에 발생한다.

    5. 카드 발동 흐름

    액션 버튼을 누른 시점부터 핸드 교체 완료까지의 처리 순서를 정의한다.
    전투 전체 턴 구조에서 이 섹션은 "액션 버튼 처리" 단계에 해당한다. (docs/systems/combat.md §액션 버튼 처리 흐름과 연동)

    5-1. 전체 처리 순서

    [액션 버튼 입력]
    │
    ▼
(1) 중복 숫자 카드 선택 처리 (§5-2)
    │
    ▼
(2) 핸드 전체 매칭 검사 (§5-3)
    │
    ├─ 매칭된 카드: 타일 소모 → 효과 실행 목록에 추가
    └─ 미매칭 카드: 효과 없이 대기
    │
    ▼
(3) 효과 실행 (§5-4) — 매칭된 카드 목록 순서대로
    │
    ▼
(4) 핸드 전체 소모덱으로 이동 (§5-5)
    │
    ▼
(5) 재드로우 — drawCards(5) 호출 (§4-2)
    │
    ▼
[다음 단계: 적 카운트 처리 → combat.md로 위임]

    5-2. 중복 숫자 카드 선택

    핸드에 동일한 effectiveTriggerNumber를 가진 카드가 여러 장 존재하고, 보드에 해당 숫자 타일이 존재할 때 선택이 필요하다.

    [미결 B-3] 중복 숫자 카드 선택 UI의 발동 시점이 확정되지 않았습니다.
    - (a) 액션 버튼을 누르기 전에 핸드 영역에서 항상 원하는 카드를 선택할 수 있는 상태 유지
    - (b) 액션 버튼을 누른 뒤 중복이 감지된 시점에 선택 팝업 표시

    (a)는 사전 계획을 강조하는 방식으로 UX가 자연스럽습니다. (b)는 구현이 단순하지만 플레이어가 액션 버튼 이후 추가 입력을 해야 하는 흐름이 됩니다.

    중복 검사 로직:

    for each uniqueTriggerNumber in hand:
    candidates = hand.filter(card => card.effectiveTriggerNumber == uniqueTriggerNumber)
    boardHasTile = board.hasTile(uniqueTriggerNumber)

    if candidates.length > 1 && boardHasTile:
        // 선택 UI 표시 → 플레이어가 1장 선택
        // 나머지 candidates는 미선택 상태가 되어 매칭에서 제외
        // (단, 보드에 같은 숫자 타일이 2개 이상이면 여러 장 발동 가능 — §5-3 참고)

    5-3. 매칭 검사

    핸드의 각 카드에 대해 보드에서 effectiveTriggerNumber와 일치하는 타일을 찾는다.

    규칙:

    • 정확한 숫자 일치만 허용한다. 더 높은 숫자 타일로 대체 발동은 불가.
    • 보드에 같은 숫자 타일이 여러 개 있으면, 핸드에 같은 숫자 카드가 여러 장일 때 각각 다른 타일을 소모하여 복수 발동이 가능하다.

    타일 소모 위치 선택:

    [미결 B-2] 보드에 같은 숫자 타일이 여러 개일 때 소모할 타일 위치 결정 방식이 확정되지 않았습니다.
    - (a) 자동 선택 — 보드 좌상단 → 우하단 방향으로 첫 번째 해당 타일을 소모
    - (b) 플레이어가 직접 소모할 타일 위치를 선택 (추가 선택 UI 필요)

    (b)는 전략적 깊이를 높이지만 조작 복잡도가 증가합니다.

    매칭 후 타일은 즉시 보드에서 제거한다. 이후 카드 효과 실행(§5-4) 순서와 관계없이, 타일 소모는 매칭 검사 완료 시점에 일괄 처리한다.

    5-4. 효과 실행 순서

    [미결 B-1] 매칭된 카드의 효과 실행 순서가 확정되지 않았습니다.
    - (a) 핸드에서 낮은 인덱스(왼쪽) 순서대로 실행
    - (b) effectiveTriggerNumber 오름차순 (낮은 숫자 먼저)
    - (c) effectiveTriggerNumber 내림차순 (높은 숫자 먼저)
    - (d) 플레이어가 실행 순서를 직접 지정

    파티 연계 발동(§5-5)의 판정이 실행 순서에 의존하므로, B-1과 연계 판정 시점(미결 B-6)을 함께 결정해야 합니다.

    효과 실행 중 팀 체력이 0이 되어도 현재 카드의 효과는 완전히 적용된 후 전투 결과를 판정한다.

    파티 연계 발동 판정:

    [미결 B-6] 연계 판정 시점이 확정되지 않았습니다.
    - (a) 카드 효과를 순차 실행하면서 직전 카드와 현재 카드의 소유 캐릭터가 다르면 연계로 판정
    - (b) 플레이어가 카드 실행 순서를 지정한 시점에 미리 연계 여부를 계산

    연계 보너스 수치는 미결 B-5 참고. 연계 발동 규칙 상세는 docs/systems/party.md (작성 예정)에서 다룬다.

    5-5. 미매칭 카드 처리

    매칭에 실패한 카드(보드에 해당 타일 없음)는 효과를 발동하지 않는다. 이 카드들은 매칭된 카드와 함께 핸드 교체 시 소모덱으로 이동한다.

    핸드 교체 처리:

    1. hand 내 모든 카드(매칭·미매칭 모두 포함)를 discardDeck으로 이동
2. hand를 빈 배열로 초기화
3. drawCards(5) 호출 (§4-2)

    6. 카드 보상 및 덱 관리

    7-1. 전투 보상 흐름

    전투 승리 시 카드 보상 선택 화면이 표시된다.

    전투 승리 확정
    ↓
보상 후보 카드 3장 무작위 선정 (보상 풀에서 추출)
    ↓
보상 화면 표시: 카드 3장을 나란히 표시, 각 카드의 이름·발동 숫자·효과 표시
    ↓
플레이어가 1장 선택
    ↓
선택한 카드를 현재 덱(waitDeck 또는 discardDeck)에 추가
    └─ 추가 위치: discardDeck 끝에 추가 (즉시 핸드에 올라오지 않음)
    ↓
보상 화면 종료 → 거점 귀환

    보상 후보 카드 3장은 현재 덱에 포함된 카드와 중복될 수 있다. 보상 화면에서 카드를 선택하지 않고 건너뛰는 옵션은 별도로 결정 필요 (현재 미정의).

    7-2. 챕터 클리어 시 덱 초기화

    챕터 클리어 확정 시점에 아래 순서로 초기화한다.

    1. DeckState 전체(waitDeck, hand, discardDeck) 소멸
2. 전투 보상으로 획득한 카드 전부 제거
3. 강화된 카드(upgradeLevel >= 1)도 제거
4. 챕터 시작 이전의 기본 덱 구성으로 복원

    기본 덱의 정의(캐릭터별 초기 카드 목록, 초기 장수)는 docs/characters.md (작성 예정)에서 관리한다.

    7-3. 거점에서의 덱 편집

    거점 귀환 후 다음 챕터 진입 전, 플레이어는 덱 편집 화면에서 아래 작업을 수행할 수 있다.

    작업 가능 여부 비고
    카드 강화 (합성) 미정
    카드 제거 미정 덱에서 특정 카드를 의도적으로 제거하는 기능 제공 여부 미결
    카드 순서 변경 미정 대기덱 순서 고정 여부 미결 (셔플이므로 의미 없을 수 있음)
