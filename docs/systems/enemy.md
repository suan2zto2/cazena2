## 1. 개요

적 AI 시스템은 **행동 카운트 기반의 완전 공개형 예측 구조**다. 모든 적은 자신의 다음 행동을 항상 플레이어에게 공개하며, 플레이어가 액션 버튼을 누를 때마다 카운트가 1씩 감소한다. 카운트가 0이 되는 순간 예고된 행동이 즉시 실행된다.

이 구조의 핵심 설계 의도는 **완전한 정보 제공 + 자원 압박**의 조합이다. 플레이어는 "오크 마법사가 1턴 후 전체 12 피해를 입힌다"는 사실을 알고 있으면서도, 남은 슬라이드 횟수와 핸드 카드 상황 때문에 완벽하게 대응하지 못하는 긴장 구조가 만들어진다.

```
[ 전투 화면 적 목록 예시 ]

고블린 검사    ▶ 3   [단일 공격 8]
오크 마법사   ▶ 1   [전체 공격 12]  ← 다음 턴 공격!
보스 드래곤   ▶ 5   [전체 공격 20 + 방어력 감소 2턴]
```

카운트 값, 행동 유형, 피해·효과 수치, 공격 대상(단일/전체)이 모두 공개된다. 숨겨진 정보는 없다.

## 2. 적 데이터 구조

### 2-1. Enemy 구조체

| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | `string` | 적 고유 식별자. 예: `"goblin_swordsman_01"` |
| `displayName` | `string` | 화면에 표시되는 이름. 예: `"고블린 검사"` |
| `type` | `EnemyType` | 적 유형. `NORMAL` / `ELITE` / `BOSS` |
| `maxHp` | `int` | 최대 체력 |
| `currentHp` | `int` | 현재 체력. 0 이하가 되면 사망 처리 |
| `shield` | `int` | 현재 방어막 수치. 피해 적용 시 체력보다 먼저 소모됨 |
| `actionCount` | `int` | 현재 행동 카운트. 액션 버튼 1회마다 -1 |
| `actionPattern` | `ActionPattern` | 행동 패턴 정의. 구조는 §2-3 및 §4 참고 |
| `currentActionIndex` | `int` | 현재 실행 예정인 행동의 인덱스 (고정 순환 방식에서 사용) |
| `nextAction` | `EnemyAction` | 예고 중인 다음 행동. 카운트 0 도달 시 이 행동을 실행 |
| `statusEffects` | `StatusEffect[]` | 현재 적용된 버프·디버프 목록 |
| `phase` | `int` | 현재 페이즈 번호. 보스 전용. 기본값 `1` |
| `phaseThresholds` | `PhaseThreshold[]` | 페이즈 전환 조건 목록. 보스 전용. §6 참고 |

**사망 판정**: `currentHp <= 0`이 되는 즉시 해당 적의 `actionCount`와 `nextAction`을 제거하고 전장에서 제외한다.

### 2-2. EnemyAction 구조체

| 필드 | 타입 | 설명 |
|------|------|------|
| `actionType` | `ActionType` | 행동 유형. `ATTACK_SINGLE` / `ATTACK_AOE` / `DEBUFF` / `BUFF_SELF` |
| `targetMode` | `TargetMode` | 대상 선택 방식. §5 행동 유형 정의 참고 |
| `power` | `int` | 피해량 또는 효과 수치의 기본값 |
| `effectId` | `string` | 디버프·버프 유형 식별자. 공격 행동에서는 `null` |
| `effectDuration` | `int` | 효과 지속 턴 수. 지속 방식에 따라 해석 방법이 달라짐 — §5-3 참고 |
| `resetCount` | `int` | 이 행동 실행 후 카운트를 리셋할 값. §3-3 참고 |
| `scheduledTurns` | `int[]` | 이 행동이 발동되는 턴 번호 목록. 예: `[3, 6, 9]`. 빈 배열이면 기본 행동 레퍼토리에서 선택 |

**`scheduledTurns` 동작**: 각 적에게 등록된 스킬 행동은 `scheduledTurns`에 지정된 턴(액션 버튼 누적 횟수 기준)에만 발동된다. 해당 턴에 발동할 스킬이 없으면 `actionPattern`에 정의된 기본 행동(통상 단일 공격)이 발동된다.

### 2-3. ActionPattern 구조체

| 필드 | 타입 | 설명 |
|------|------|------|
| `patternType` | `PatternType` | 패턴 유형. `FIXED_CYCLE` / `RANDOM_WEIGHTED` / `CONDITIONAL` — §4 참고 |
| `actions` | `EnemyAction[]` | 행동 목록. 패턴 유형에 따라 사용 방식이 다름 |
| `defaultAction` | `EnemyAction` | 해당 턴에 스킬이 없을 때 발동되는 기본 행동 (통상 단일 공격) |
| `initialCount` | `int` | 전투 시작 및 리셋 시 카운트 기본값 |

### 2-4. 적 유형별 특성

| 유형 | `EnemyType` | 카운트 초기값 범위 | 행동 레퍼토리 특성 |
|------|------------|------------------|-------------------|
| 일반 | `NORMAL` | 2 ~ 4 | 단일 공격 위주. 디버프 없음 또는 1종만 보유 |
| 엘리트 | `ELITE` | 2 ~ 3 | 단일·광역 공격 혼재. 디버프·자기 강화 보유 가능 |
| 보스 | `BOSS` | 3 ~ 5 (페이즈에 따라 변동) | 다수의 행동 유형, 페이즈별 패턴 교체. 반드시 광역 공격 포함 |

> **[미결 C-1]** 카운트 리셋값 방식이 미결 상태다. 선택지는 다음과 같다.
> - **(a) 고정 초기값 리셋**: `actionPattern.initialCount`로 항상 리셋. 구현 단순, 적 행동 주기가 일정하여 플레이어 예측이 용이함.
> - **(b) 행동별 가변 리셋값**: 발동된 `EnemyAction.resetCount` 값으로 리셋. 강력한 행동은 긴 쿨타임, 약한 행동은 짧은 쿨타임을 표현할 수 있음. 구현 복잡도 증가.
>
> 위 표의 카운트 초기값 범위는 (a) 기준으로 작성되었다. (b) 채택 시 `EnemyAction.resetCount` 필드를 모든 행동 데이터에 필수값으로 설정해야 하며, `ActionPattern.initialCount`는 전투 시작 시 최초 카운트에만 사용한다.

## 3. 행동 카운트 처리 규칙

### 3-1. 카운트 감소 타이밍

```
플레이어 액션 버튼 누름
    ↓
(1) 핸드 카드 발동 처리 완료
(2) 핸드 전체 소모덱 이동 + 재드로우
(3) 전장의 모든 살아있는 적 카운트 -1  ← 이 시점
(4) 카운트 0인 적 행동 처리
```

- 슬라이드는 카운트에 영향을 주지 않는다. 오직 **액션 버튼 1회 = 모든 적 카운트 -1**이다.
- 카운트 감소는 카드 발동이 완전히 끝난 후 일괄 처리된다. 카드 발동 도중에 카운트가 변하지 않는다.
- 사망한 적(currentHp <= 0)은 카운트 감소 대상에서 제외된다.

### 3-2. 카운트 0 도달 시 처리 흐름

```
카운트 0 감지 (전체 적 순회 후)
    ↓
카운트 0인 적이 한 명인 경우: 즉시 nextAction 실행
카운트 0인 적이 둘 이상인 경우: 적 목록 위→아래 순서로 순차 실행
    ↓
행동 실행 전 처치 확인: currentHp <= 0이면 행동 취소 (§7-1 참고)
    ↓
nextAction 실행
    ↓
카운트 리셋 (§3-3 참고)
    ↓
다음 nextAction 결정 및 예고 UI 갱신
```

**복수 적 동시 카운트 0 처리 예시**:

전장에 적 A(위), 적 B(아래)가 동시에 카운트 0이 된 경우:
1. 적 A의 `nextAction` 실행 → 적 A 카운트 리셋 → 적 A 다음 예고 등록
2. 적 B의 `nextAction` 실행 → 적 B 카운트 리셋 → 적 B 다음 예고 등록

적 A의 행동으로 적 B가 사망하는 상황은 발생하지 않는다(적이 아군을 공격하므로). 단, 적 A의 행동으로 아군 캐릭터가 사망하더라도 적 B의 행동은 계속 실행된다.

### 3-3. 카운트 리셋값 결정

> **[미결 C-1]** §2-4의 미결 사항과 동일. 확정 전에는 **(a) 고정 초기값 리셋** 기준으로 구현한다.

**(a) 고정 초기값 리셋 기준 구현**:

```
행동 실행 완료
    ↓
actionCount = actionPattern.initialCount
```

**(b) 행동별 가변 리셋값 기준 구현** (C-1이 (b)로 결정된 경우):

```
행동 실행 완료
    ↓
actionCount = 실행된 EnemyAction.resetCount
```

## 4. 행동 패턴 구조

> **[미결 C-2]** 행동 패턴 구조가 미결 상태다. 선택지는 다음과 같다.
> - **(a) 고정 순환 (`FIXED_CYCLE`)**: `actions` 배열을 인덱스 순서대로 반복. 1→2→3→1→2→3. 구현 단순, 플레이어 예측이 완전히 가능하여 보스에 적용 시 전략성이 높아짐.
> - **(b) 랜덤 가중치 선택 (`RANDOM_WEIGHTED`)**: `actions` 배열의 각 항목에 `weight` 값을 부여하고, 가중치 비율로 무작위 선택. 예측 불가능성 부여 가능. 별도 `weight` 필드 추가 필요.
> - **(c) 조건 분기 (`CONDITIONAL`)**: 현재 체력 비율, 현재 카운트값, 상태이상 보유 여부 등 조건에 따라 발동 행동을 결정. 가장 풍부한 기믹 설계 가능. 구현 복잡도 가장 높음.
>
> 확정 전에는 **(a) 고정 순환** 기준으로 구현한다. 이 경우 `currentActionIndex`가 매 행동 실행마다 +1 증가하며, `actions` 배열 끝에 도달하면 0으로 순환한다.

### 4-1. 고정 순환 방식 (a) 기준 처리 흐름

```
전투 시작
    ↓
currentActionIndex = 0
nextAction = actions[0]  (첫 번째 예고 등록)
    ↓
카운트 0 도달 → actions[currentActionIndex] 실행
    ↓
currentActionIndex = (currentActionIndex + 1) % actions.length
nextAction = actions[currentActionIndex]  (다음 예고 등록)
    ↓
카운트 리셋
```

단, `scheduledTurns`에 지정된 스킬 행동이 있을 경우 순환 인덱스보다 스킬 발동이 우선된다. §4-3 참고.

### 4-2. 스킬 행동과 기본 행동의 관계

각 적은 `actionPattern.actions`에 기본 행동 레퍼토리를 가지며, 별도로 `scheduledTurns`가 지정된 스킬 행동이 등록될 수 있다.

```
액션 버튼 누적 횟수(globalTurnCount) 확인
    ↓
현재 turnCount에 발동할 scheduledTurns 스킬이 있는가?
    ├─ 있음: 해당 스킬 행동을 nextAction으로 예고
    └─ 없음: actionPattern의 기본 행동(defaultAction 또는 순환 인덱스)을 nextAction으로 예고
```

**예시 — 오크 마법사 행동 데이터**:

```
actions (기본 순환):
  [0] 단일 공격 8 (resetCount: 3)
  [1] 단일 공격 8 (resetCount: 3)
  [2] 전체 공격 12 (resetCount: 3)

스킬 행동:
  scheduledTurns: [5, 10, 15] → 방어력 감소 디버프 (전체)
```

- 턴 1~4: 기본 순환 진행 (단일 공격 8 → 단일 공격 8 → 전체 공격 12 → 단일 공격 8 ...)
- 턴 5: 스킬 발동 → 방어력 감소 디버프 (전체) 예고 등록
- 턴 6~9: 기본 순환 재개

### 4-3. 같은 턴 여러 스킬 겹침 처리

> **[미결 C-3]** 같은 턴에 발동할 스킬이 두 개 이상 겹치는 경우의 처리 방식이 미결 상태다.
> - **(a) 등록 순서대로 모두 실행**: 복수 스킬을 순서대로 모두 발동. 강력한 조합 기믹 설계 가능. 플레이어에게 복수 예고를 동시 표시해야 함.
> - **(b) 첫 번째 스킬만 실행**: `scheduledTurns` 배열의 앞 순서 스킬만 발동. 구현 단순. 나머지 스킬은 해당 턴 무시됨.
> - **(c) 합산 처리**: 겹치는 스킬의 수치를 합산하여 하나의 행동으로 처리. 예고 UI에 합산 수치를 표시.
>
> 확정 전에는 **(b) 첫 번째 스킬만 실행** 기준으로 구현한다. 데이터 작성 시 같은 `scheduledTurns` 값을 가진 스킬이 겹치지 않도록 설계한다.

## 5. 행동 유형 정의

### 5-1. 단일 공격 (`ATTACK_SINGLE`)

출전 캐릭터에게 피해를 입힌다.

| 파라미터 | 필드 | 설명 |
|----------|------|------|
| 피해량 | `power` | 적용할 기본 피해 수치 |
| 대상 선택 | `targetMode` | 1인 전투이므로 항상 출전 캐릭터 |

**피해 적용 순서**:
1. 캐릭터의 `shield > 0`이면 피해를 `shield`에서 먼저 차감
2. `shield`가 0이 되면 남은 피해를 캐릭터 체력에서 차감
3. 캐릭터 체력이 0 이하가 되면 전투 패배 판정

> **[미결 C-6]** 단일 공격의 대상 선택 방식이 미결 상태다.
> - **(a) 항상 첫 번째 생존 파티원**: 파티원 목록의 가장 앞 순서(인덱스 0번부터)에서 생존한 캐릭터를 선택. 구현 최단순. 플레이어가 대상을 완전히 예측 가능.
> - **(b) 체력이 가장 낮은 파티원**: 현재 `currentHp`가 가장 낮은 생존 캐릭터를 선택. 집중 타격 AI처럼 동작. 동점 시 인덱스 앞 순서 선택.
> - **(c) 무작위 파티원**: 생존한 파티원 중 균등 무작위 선택. 예측 불가능성 부여.
> - **(d) 적 패턴 데이터에 명시**: `EnemyAction`에 `targetMode` 필드로 적마다 다른 타겟팅 방식을 지정. 가장 풍부한 설계 가능. 구현 복잡도 증가.
>
> 확정 전에는 **(a) 첫 번째 생존 파티원** 기준으로 구현한다. (d) 채택 시 `TargetMode` 열거형에 `FIRST` / `LOWEST_HP` / `RANDOM`을 추가한다.

### 5-2. 광역 공격 (`ATTACK_AOE`)

출전 캐릭터에게 피해를 입힌다. 1인 전투이므로 단일 공격과 동일하게 동작한다.

| 파라미터 | 필드 | 설명 |
|----------|------|------|
| 피해량 | `power` | 캐릭터에게 적용할 기본 피해 수치 |
| 대상 | — | 항상 출전 캐릭터. `targetMode` 무시 |

**피해 적용 순서**: §5-1과 동일한 방어막 우선 소모 규칙이 적용된다.

### 5-3. 디버프 (`DEBUFF`)

파티원 1명 또는 전체에게 부정적 상태이상을 부여한다.

| 파라미터 | 필드 | 설명 |
|----------|------|------|
| 효과 식별자 | `effectId` | 부여할 디버프 종류. 예: `"attack_down"`, `"defense_down"`, `"poison"` |
| 효과 수치 | `power` | 디버프 강도. 예: 공격력 감소 값, 독 피해량 |
| 지속 방식 | `effectDuration` | 지속 턴 수 또는 영구 여부 결정값 — 아래 참고 |
| 대상 | `targetMode` | `SINGLE` (C-6 미결 방식 적용) 또는 `ALL` |

> **[미결 C-5]** 디버프·버프의 지속 방식이 미결 상태다.
> - **(a) N턴 지속 (액션 버튼 횟수 기준)**: `effectDuration`에 지속 턴 수를 저장. 플레이어가 액션 버튼을 누를 때마다 -1. 0이 되면 자동 해제. 가장 일반적인 방식. `StatusEffect` 구조체에 `remainingDuration` 필드 필요.
> - **(b) 영구 지속**: `effectDuration` 값과 무관하게 해제 스킬이나 특정 조건 없이는 유지됨. 디버프 해제 카드나 기믹이 반드시 필요함.
>
> 확정 전에는 **(a) N턴 지속** 기준으로 구현한다. `effectDuration = 0`은 영구 지속의 특수값으로 예약해둔다.

**기본 디버프 종류 (초안)**:

| effectId | 효과 설명 | power 해석 |
|----------|-----------|------------|
| `attack_down` | 카드 발동 피해량 감소 | 감소량 (절대값) |
| `defense_down` | 방어막 획득량 감소 | 감소율 (%) |
| `poison` | 액션 버튼마다 체력 감소 | 감소량 |
| `count_reduce` | 해당 파티원 카드 발동 불가 | 발동 불가 카드 수 |

### 5-4. 자기 강화 버프 (`BUFF_SELF`)

적 자신에게 유리한 상태를 부여한다.

| 파라미터 | 필드 | 설명 |
|----------|------|------|
| 효과 식별자 | `effectId` | 부여할 버프 종류. 예: `"shield_gain"`, `"attack_up"` |
| 효과 수치 | `power` | 버프 강도. 예: 방어막 획득량, 공격력 증가 값 |
| 지속 방식 | `effectDuration` | §5-3 C-5 미결 사항과 동일 규칙 적용 |
| 대상 | — | 항상 자기 자신. `targetMode` 무시 |

**기본 버프 종류 (초안)**:

| effectId | 효과 설명 | power 해석 |
|----------|-----------|------------|
| `shield_gain` | 방어막 부여 | 방어막 수치 |
| `attack_up` | 다음 공격 피해량 증가 | 증가량 (절대값) |
| `count_reset` | 자신의 카운트를 즉시 리셋 | 리셋 후 카운트값 |

## 6. 보스 페이즈 시스템

보스(`BOSS` 유형)는 체력 구간에 따라 행동 패턴이 교체된다. 패턴 교체 시 카운트가 리셋되고 새 예고가 즉시 등록된다.

### 6-1. PhaseThreshold 구조체

| 필드 | 타입 | 설명 |
|------|------|------|
| `phaseNumber` | `int` | 전환될 페이즈 번호. 1이 초기 페이즈 |
| `triggerValue` | `number` | 전환 트리거 값. C-4 방식에 따라 해석이 다름 (§6-2 참고) |
| `actionPattern` | `ActionPattern` | 해당 페이즈에서 사용할 행동 패턴 |
| `transitionAction` | `EnemyAction?` | 페이즈 전환 시 즉시 발동하는 특수 행동. `null`이면 전환 행동 없음 |

> **[미결 C-4]** 보스 페이즈 전환 트리거 정의 방식이 미결 상태다.
> - **(a) 체력 비율**: `triggerValue`를 0.0~1.0의 체력 비율로 해석. 예: `0.5`이면 최대 체력의 50% 이하가 될 때 전환. 보스 체력 수치에 무관하게 데이터 작성 가능.
> - **(b) 절대 체력값**: `triggerValue`를 절대 체력값으로 해석. 예: `500`이면 현재 체력이 500 이하가 될 때 전환. 체력값에 직접 대응하여 기획 의도가 명확함.
>
> 확정 전에는 **(a) 체력 비율** 기준으로 구현한다. `triggerValue` 타입은 `float` (0.0 ~ 1.0).

**페이즈 전환 조건 판정 시점**: 적의 `currentHp`가 변경될 때마다(피해 적용 후) 즉시 확인한다.

### 6-2. 페이즈 전환 처리 순서

```
피해 적용으로 currentHp 감소
    ↓
phaseThresholds 확인: currentHp / maxHp <= triggerValue 인 미전환 PhaseThreshold 존재?
    ├─ 없음: 계속 진행
    └─ 있음: 페이즈 전환 처리 시작
               ↓
           phase = phaseThreshold.phaseNumber
               ↓
           actionPattern = phaseThreshold.actionPattern
           currentActionIndex = 0
               ↓
           transitionAction != null 이면: transitionAction 즉시 실행
               ↓
           actionCount = 새 actionPattern.initialCount  (카운트 리셋)
               ↓
           nextAction = 새 패턴의 첫 번째 행동  (예고 갱신)
```

**페이즈 전환 중 행동 취소 규칙**: 페이즈 전환이 발생하는 시점에 기존 `nextAction`(이전 패턴의 예고)은 즉시 폐기된다. 전환 트리거가 된 피해 적용 과정에서 카운트가 0이었더라도, 페이즈 전환이 우선 처리되어 기존 행동 대신 `transitionAction`이 실행된다.

**보스 페이즈 데이터 예시 (드래곤 보스)**:

```
phase 1 (체력 100% ~ 50%):
  initialCount: 4
  actions: [단일 공격 20, 전체 공격 15, 단일 공격 20, 방어막 40 자기 강화]
  defaultAction: 단일 공격 20

phaseThreshold [1 → 2]:
  triggerValue: 0.5  (체력 50% 이하)
  transitionAction: 전체 공격 30 (페이즈 전환 연출)
  
phase 2 (체력 50% ~ 25%):
  initialCount: 3
  actions: [전체 공격 20, 전체 공격 20, 전체 공격 20 + 방어력 감소]
  defaultAction: 전체 공격 20

phaseThreshold [2 → 3]:
  triggerValue: 0.25  (체력 25% 이하)
  transitionAction: null

phase 3 (체력 25% ~):
  initialCount: 2
  actions: [전체 공격 25, 전체 공격 30]
  defaultAction: 전체 공격 25
```

## 7. 예외 처리

### 7-1. 카운트 0 + 동시 처치 시 행동 취소

카드 발동으로 적을 처치하는 것과 해당 적의 카운트가 0이 되는 것이 같은 액션 버튼 처리 내에서 동시에 발생할 경우:

```
처리 순서:
(1) 카드 효과 적용 (피해)
(2) currentHp <= 0 확인 → 사망 처리 (actionCount 및 nextAction 즉시 제거)
(3) 전체 적 카운트 -1
(4) 카운트 0인 적 행동 처리 단계: 해당 적은 이미 사망이므로 목록에 없음 → 행동 실행 없음
```

행동 취소가 적용되는 시나리오: 적의 카운트가 1이었을 때 플레이어가 액션 버튼을 누르면 카드 발동 후 카운트가 0이 된다. 만약 그 카드 발동으로 적이 사망했다면 행동은 취소된다.

### 7-2. 전체 공격 시 방어막 우선 소모

광역 공격(`ATTACK_AOE`) 및 전체 대상 디버프 적용 시:
- 각 파티원의 `shield` 값을 피해보다 먼저 소모한다.
- `shield`가 피해량보다 크면 체력에 피해 없이 방어막만 감소한다.
- `shield`가 피해량보다 작으면 방어막을 0으로 만든 뒤 초과 피해를 `currentHp`에 적용한다.

단일 공격(`ATTACK_SINGLE`)도 동일한 방어막 우선 소모 규칙을 따른다.

### 7-3. 적 사망 시 즉시 제거

적의 `currentHp`가 0 이하가 되는 즉시:
- 해당 적의 `actionCount`를 초기화하고 예고 UI에서 제거한다.
- 해당 적의 `nextAction` 데이터를 무효화한다.
- 이후 모든 카운트 처리 순회에서 해당 적은 건너뛴다.
- 남은 액션 버튼 처리가 계속 진행되어 다른 적의 카운트 감소 및 행동 처리는 정상 진행된다.

### 7-4. 보스 페이즈 전환 중 카운트 0 겹침

피해 적용으로 페이즈 전환이 발생하는 동시에 카운트가 0이 된 경우(예: 이전 페이즈에서 카운트가 이미 1이었던 상태에서 피해 → 카운트 -1로 0이 되면서 동시에 페이즈 전환):

```
처리 우선순위:
(1) 페이즈 전환 처리 (이전 nextAction 폐기, 새 패턴 적용, transitionAction 실행)
(2) 카운트 리셋 (새 actionPattern.initialCount로 리셋)
(3) 이전 nextAction 행동은 실행하지 않음
```

즉, 페이즈 전환이 카운트 0 행동보다 항상 우선한다.

### 7-5. 상태이상 중첩

동일한 `effectId`의 디버프·버프가 이미 적용된 상태에서 다시 부여될 경우:
- **지속 시간 갱신**: 기존 효과의 `remainingDuration`을 새 `effectDuration` 값으로 덮어씀. 중첩 누적은 하지 않는다.
- **수치 갱신**: 기존 효과의 `power` 값을 새 `power` 값으로 덮어씀.

단, 서로 다른 `effectId`의 효과는 독립적으로 중첩 적용된다.
